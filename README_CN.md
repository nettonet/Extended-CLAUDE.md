# 增强版 CLAUDE.md — 对 andrej-karpathy-skills 的补充

本仓库提供了一个**增强版 `CLAUDE.md`**，它直接基于
**[forrestchang/andrej-karpathy-skills](https://github.com/forrestchang/andrej-karpathy-skills)**
（原始 Karpathy 风格的 LLM 编码智能体行为准则）构建。

## 原项目涵盖的内容

原 `CLAUDE.md` 提炼了来自 Andrej Karpathy 对 LLM 编码失败观察的四条久经考验的原则：

| 原则 | 核心思想 |
|---|---|
| **编码前先思考** | 明确假设，处理混淆，呈现权衡 |
| **简单优先** | 最少代码，无投机性抽象 |
| **外科手术式修改** | 只接触必需部分，匹配现有风格 |
| **目标驱动执行** | 定义可验证的成功标准，循环直到通过 |

这四条规则在减少不必要的差异、防止过度工程化以及将 AI 编码智能体转变为谨慎协作伙伴（而非鲁莽打字员）方面，已证明其显著有效性。

## 为什么我们要扩展它

原指南刻意保持简洁（约 70 行）。在实际使用中，我们发现三种反复出现的模式值得明确地行为强化。我们的扩展将这些作为**独立的新原则**加入，同时保持原四条完全不变。

### 我们新增了什么（以及为什么）

#### 1. 停止过度思考——“三思而后行，再思可矣”

原 `Think Before Coding` 鼓励谨慎推理，但 LLM（特别是 Claude Code）可能会陷入**分析瘫痪**——生成多个几乎相同的解决方案，反复评估相同的权衡，并在实施前猜测上耗费大量 token。

**我们的补充：** 一个硬性停止点。一旦出现清晰、简单的方向，就选择一种方法并继续。根据具体反馈（编译器错误、测试结果、用户反应）进行改进，而不是基于假设性推测。

#### 2. 核心优先，边缘情况靠后——“众星朗朗，不如孤月独明”

原 `Simplicity First` 警告不要添加投机性功能和抽象，但没有明确解决*优先级倒置*的问题——在主流程尚未稳固之前，边缘情况处理就膨胀了复杂性。

**我们的补充：** 在打磨罕见分支之前，让核心路径变得坚如磐石。一个精心处理的边缘情况永远无法挽救一个失败的主要工作流。当资源（注意力、上下文窗口、时间）有限时，将它们投入到影响最大的地方。

#### 3. 容忍不完美——“瓜无滚圆，人无十全”

原原则倾向于*理论上的正确性*。真实的代码存在于一个混乱的世界中：浮点运算、传感器噪声、模型置信度分数、人为拼写错误。为几乎不可能发生的场景追求完美恢复，会使复杂性与价值不成比例地爆炸。

**我们的补充：** 对容差、阈值、优雅降级的明确指导，以及务实接受“几乎从不失败”是一个可接受的工程答案。

### 理念
这些扩展与原版共享相同的设计理念：
可操作，而非哲学。 每条规则都会引发智能体可观察的行为变化。
可测试。 通过检查差异、PR 和交互日志，你可以判断规则是否在起作用。
偏向谨慎，但有逃生舱。 指南倾向于谨慎行为，但明确允许对于琐碎任务进行判断。
